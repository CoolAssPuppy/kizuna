## Executive summary

1. **The eight largest user-facing screens have zero Vitest coverage**, despite CLAUDE.md/AGENTS.md mandating "TDD non-negotiable. Components get Vitest tests first." Only 6 `*.test.tsx` files exist in `src/` against 80+ `*.test.ts` for utilities. The TDD invariant is being honoured for libs, not for components. (F-T01)
2. **`AgendaAdminScreen.tsx` (615 LOC) holds two distinct screens** — the live-session list and the proposals review queue — switched by a single `dayFilter === PROPOSED_FILTER` flag. Each has its own data, mutations, and rendering branch. Split. (F-A01)
3. **`@vitest/ui` is in `dependencies`, not `devDependencies`** (`package.json:50`). It ships with the production bundle. One-line fix; very high payoff. (F-D01)
4. **`zod-to-json-schema` is declared in root deps but never imported anywhere.** (`package.json:71`) Drop it. (F-D02)
5. **Documentation drift on table count and locale support.** README says "37 tables" (line 291), CLAUDE.md says "35 tables" — actual count via `grep -c "create table"` in `supabase/schemas/` is **48**. README also claims "English-US default and only locale today" (line 387) but `src/lib/i18n.ts:14-23` loads 8 locales; `de-DE` has 892 keys vs `en-US`'s 1230 (27% gap), implying half-translated locales the README doesn't acknowledge. (F-Doc01, F-Doc02)
6. **`useEditorialFeed` issues an unbounded `select * from feed_items` for the active event** (`src/features/home/useEditorialFeed.ts:23-28`). RLS narrows it for non-admins to display-window items, but admins get the full set, and there is no `.limit()`. Add a cap before the home screen lands in front of admins post-event. (F-P01)
7. **PostgREST `.or()` filters interpolate user-controlled strings unescaped** in `src/features/community/photos/api.ts:178` (`email.ilike.%${query.value}%`). A comma in `query.value` splits the filter into multiple clauses; RLS still gates the rows but the search semantics break. Escape commas, parens, dots before interpolation. (F-S01)
8. **CORS `Access-Control-Allow-Origin: *` is shipped on every edge function** (`supabase/functions/_shared/cors.ts:13`). Already self-documented as a hardening item; schedule a tightening to the production hostname before launch. (F-S02)
9. **Native `confirm()` / `window.confirm()` is used in 8 admin/profile flows** for destructive actions (`AgendaAdminScreen.tsx:390`, `EventEditScreen.tsx:323`, `DocumentsScreen.tsx:111`, `FeedScreen.tsx:180`, `TagsDialog.tsx:143`, `SwagAdminScreen.tsx:111`, `ApiKeysSection.tsx:50`, `PhotoLightbox.tsx:68`). Jarring against an otherwise polished shadcn UI; reviewers will notice on day one. (F-C01)
10. **Twenty-two `as unknown as` casts in `src/`**, almost all in `features/*/api/*.ts` to coerce join shapes that Supabase typegen resolves as `T | T[] | null`. The `flatJoin` helper in `src/lib/supabase.ts:44` solves it for some sites; extend it (or generate stricter view types) and remove the casts. None are dangerous; all are noise that signals "the types fight us." (F-T02)

## Architectural mental model

Kizuna is a Vite + React 18 PWA whose single source of truth is a Postgres schema declared file-by-file in `supabase/schemas/`. The web tier is feature-sliced (`src/features/<name>/{api,hooks,components}`) with a small `src/lib/` for cross-cutting concerns and `src/components/` for shared UI. State flows through TanStack Query against a singleton anon Supabase client (`src/lib/supabase.ts`); no Redux/Zustand at scale. Mutations route through `useMutation`; long-running or privileged work goes through edge functions wrapped by `callEdgeFunction<T>` (`src/lib/edgeFunction.ts`). Realtime is used surgically: 5 sites total (`messages`, `itinerary`, `notifications`, typing presence, plus a generic `useRealtimeInvalidation`), each with explicit `removeChannel` cleanup.

The security perimeter is RLS-on-every-table enforced by a `do $$ ... loop ... execute format ...` block at `90_rls.sql:145-158`, plus 36+ SECURITY DEFINER helpers, all of which set `search_path = public` and are individually granted (only the read-only helpers — `is_admin`, `auth_role`, `is_self_or_admin`, etc. — are EXECUTE-able by `anon`/`authenticated`; everything else is `service_role`-only or invoked through user-side RPC). The custom-claim `app_role` lives in JWT alongside the standard Supabase `role` (`employee | guest | admin | super_admin`); RLS reads it through `public.auth_role()`. Service role exists only inside edge functions (`supabase/functions/_shared/supabaseClient.ts:27`) and two scripts (`seed-test-storage.ts:27`, `seed-remote-users.ts:59`). The client SPA uses the publishable key only.

The two friction points the architecture pushes back against:

1. **Generated Supabase types vs ergonomic queries** — every `.from(...).select('a, b, c, joined(...)')` produces a typed shape that doesn't always match what the query actually returns when there are aliased FKs or 1-to-1 joins. The codebase uses `flatJoin<T>()` and `as unknown as` to bridge. Fine for now; it's friction the typegen team upstream is working on.
2. **The "one big screen with tabs" admin pattern** vs feature-sliced ideals — `AgendaAdminScreen` and `HomeScreen` both pack 3-5 sub-features into one route. The router has no nested routes for these, so the screen-as-tab-host pattern is the workaround. It costs file size.

The README's claim "37 tables" (line 291) and CLAUDE.md's "35 tables" both predate the photo gallery, CLI api keys, OAuth codes, swag catalog, session proposals/voting/tags, profile custom fields, and dependent guests work. The actual count is 48.

## Things that look bad but are actually fine

- **22 `as unknown as` casts in api/ files (F-T02).** Almost every one is a Supabase-typegen workaround for joins where the inferred shape (`T \| T[] \| null`) doesn't match the runtime reality (always `T` or always `T[]`). The `flatJoin<T>()` helper at `src/lib/supabase.ts:44` and the inline comments at the cast sites document this. They're noise, not danger. Worth fixing eventually (F-T02), not urgent.
- **Realtime subscriptions look uncleaned (5 sites with `.subscribe()`).** Every site has a paired `removeChannel(channel)` in the effect cleanup. Verified at: `ChannelScreen.tsx:104-106`, `useTypingPresence.ts:68-71`, `useItinerary.ts:66-68`, `useNotifications.ts:58-60`, `useRealtimeInvalidation.ts:59-61`. Nothing to fix.
- **`window.addEventListener('storage', ...)` at module scope in `src/features/events/eventOverride.ts:28`** without a `removeEventListener` cleanup. This is correct: the listener lives for the page's lifetime and watches localStorage cross-tab. There's no "unmount" event for a module.
- **`schema.sql` files are huge** (`80_functions_and_triggers.sql` 1658 LOC, `90_rls.sql` 1013 LOC). Splitting helps reading but Supabase's declarative-schema tool prefers fewer larger files in some configurations, and pgTAP test naming groups by domain anyway. Cosmetic. F-A08/F-A09 flag it; not a forcing function.
- **CORS `Access-Control-Allow-Origin: *`.** Already self-documented as known debt. Bearer-JWT auth means a wide-open origin doesn't expose data; it's a defence-in-depth tightening, not a CVE. F-S02 flags it but it's not Critical.
- **No zod validation on edge function bodies (F-S04).** All 13 functions either reject anything but a single string field, or trust an integer/uuid that PostgREST will reject downstream. zod is nicer for evolution but not currently a vulnerability.
- **`.env` in `ls -la` looks scary** (F-S05) — but it's not git-tracked. Local dev only.
- **README references `TECH_DEBT_AUDIT.md` that didn't exist before this audit.** The earlier audit was either lost in a rebase or never written; the inline `(TECH_DEBT_AUDIT F008)` reference in `cors.ts:9` will need re-tagging against this audit's `F-S02`. Cosmetic, F-Doc03.
- **17 `useEffect` sites in src** despite the `no-useEffect` skill. All 17 are genuine: realtime cleanup (5), scroll/layout effects (3), one-shot module-load detection (2), debounced inputs (1), the rest are refs/intersection observers. The 36 `useMountEffect` sites are the disciplined replacements for one-time mount effects. The proportion is healthy.
- **`select('*')` 25+ times.** For most of these, the consumer reads the full row. Bloating risk only matters if a future migration adds a large blob column without auditing consumers, which is a process problem, not a code problem.
